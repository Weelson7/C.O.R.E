# C.O.R.E. Branding Philosophy

## 1. Brand Core

**C.O.R.E.** stands for **Clustered Orchestrated Repurposed Environment**.

It is not a "home lab toy" and not a "single server setup." It is a resilient digital organism built from repurposed hardware where each node is replaceable, observable, and accountable to the whole.

The brand must always communicate three things:

1. **Resilience over fragility**
2. **Clarity over noise**
3. **System thinking over machine worship**

Tagline foundation:

> The network is the computer. The cluster is the organism.

## 2. Vision

Build a self-hosted infrastructure identity that feels:

- **Industrial and alive**: robust like engineered machinery, responsive like a living system.
- **Predictable under pressure**: even during failures, the interface should communicate control, not panic.
- **Technically elegant**: no decorative complexity without operational purpose.

Long-term brand vision:

- C.O.R.E. becomes a recognizable visual language for self-hosted cluster operations.
- Every service surface should feel like part of one mesh-aware system.
- The UI should make redundancy and health legible at a glance.

## 3. Values

### Cluster-first

Every decision should present services as members of a mesh, never isolated apps.

### Operational honesty

If a service is down, show it clearly. Do not hide degraded state behind pretty visuals.

### Zero-trust mindset

Security is not optional or "advanced mode." It is baseline behavior.

### Repeatability

The design language should support scriptable deployment and consistent structure across all service pages.

### Redundancy as identity

Backup, failover, and observability are part of the product personality, not background plumbing.

## 4. Brand Personality

The C.O.R.E. voice is:

- **Precise**: concise wording, specific states, actionable labels.
- **Calm**: never frantic, even in error states.
- **Confident**: decisive language, no ambiguous UX copy.
- **Technical but human**: expert tone without unnecessary jargon.

Avoid:

- Meme language
- Vague status text ("something went wrong")
- Overly playful metaphors during incidents

## 5. Messaging Principles

### Principle A: Show system context

Always anchor service status to node context when relevant.

Example:

- Prefer: "Jellyfin down on NODE-ALPHA (10.10.0.10)."
- Avoid: "Media unavailable."

### Principle B: Distinguish incident levels

- **Info**: expected transitions and maintenance
- **Warning**: degraded but functioning
- **Critical**: outage or failover needed

### Principle C: Pair problems with next actions

Every warning/critical banner should include an obvious operator action path.

## 6. Visual Philosophy

The visual system should express:

- **Deep-space foundation** (black-violet base)
- **Neon signal layer** (purple electric accents)
- **Health telemetry** (green/yellow/red semantics)
- **Shimmering energy** (measured glow and pulse, never constant noise)

Canonical brand mark:

- Use the official logo file at `Branding/logo.png` as the primary visual identifier for C.O.R.E.
- Do not replace it with autogenerated glyphs or alternative symbols in production interfaces.

The interface should feel like:

- A secure operations deck
- Running in real time
- Built for uptime decisions

Not like:

- A gaming HUD clone
- A generic SaaS dashboard
- A loud cyberpunk poster

## 7. UX Intent by Scenario

### Normal operation

Mood: calm, luminous, readable.

- Low-motion ambient shimmer
- Soft neon borders
- Stable status indicators

### Degraded mode

Mood: caution without alarm fatigue.

- Introduce amber highlights
- Increase contrast on affected cards
- Keep layout stable; no visual jumps

### Incident / outage

Mood: urgent and actionable.

- Red-critical signaling appears fast
- Incident summary pinned near top
- Animation becomes minimal and purposeful

### Failover active

Mood: controlled transition.

- Purple remains brand anchor
- Cyan "rerouted" indicator can appear as secondary accent
- Show source node -> target node mapping clearly

## 8. Consistency Rules

1. **Same service, same semantics everywhere**
2. **Status colors are reserved for status only**
3. **Glow supports hierarchy, not decoration**
4. **Motion explains state changes, not style trends**
5. **If uncertain, choose readability and speed over visual flair**

## 9. Brand Promise

C.O.R.E. promises a cluster experience where:

- infrastructure feels intelligible,
- failures feel manageable,
- and the system remains trustworthy under stress.

That promise must be visible in every screen, every service card, and every status change.
